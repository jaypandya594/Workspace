     1Ôćĺ# iSecurify GRC Platform ÔÇö Coolify Deployment Guide
     2Ôćĺ
     3ÔćĺA full-featured Governance, Risk, and Compliance (GRC) platform for managing policies, frameworks, controls, audits, risks, vulnerabilities, evidence, and multi-tenant operations. This guide walks you through deploying the entire stack on **Coolify** ÔÇö a self-hosting PaaS that runs on your own VPS.
     4Ôćĺ
     5Ôćĺ---
     6Ôćĺ
     7Ôćĺ## Architecture Overview
     8Ôćĺ
     9Ôćĺ| Layer | Technology | Details |
    10Ôćĺ|-------|-----------|---------|
    11Ôćĺ| **Frontend** | Next.js 16 (App Router) | React 19, TypeScript, Tailwind CSS 4, shadcn/ui components |
    12Ôćĺ| **Runtime** | Bun | Used as the Node runtime (`bun .next/standalone/server.js`) |
    13Ôćĺ| **Output Mode** | Next.js Standalone | `output: "standalone"` in `next.config.ts` ÔÇö single deployable artifact |
    14Ôćĺ| **Database** | MySQL 8.0 | Coolify-managed resource with persistent Docker volume |
    15Ôćĺ| **ORM** | Prisma 6.x | Schema sync via `prisma db push` (no migration files) |
    16Ôćĺ| **Authentication** | Custom scrypt session auth | `scryptSync` password hashing, session tokens stored in DB, HttpOnly cookies |
    17Ôćĺ| **File Uploads** | Docker volume | Uploaded evidence files persisted at `/app/uploads` inside the container |
    18Ôćĺ| **API** | Next.js Route Handlers | RESTful JSON API under `/api/*` with RBAC per tenant |
    19Ôćĺ| **UI State** | Zustand + TanStack Query | Client-side state management and server-state caching |
    20Ôćĺ| **Charts** | Recharts | Dashboard analytics, risk heatmaps, audit progress |
    21Ôćĺ
    22Ôćĺ### Key Design Decisions
    23Ôćĺ
    24Ôćĺ- **Prisma `db push` instead of migrations** ÔÇö the schema is the source of truth; `prisma db push` synchronizes it directly to the database without generating migration files. This is ideal for early-stage SaaS where the schema evolves rapidly.
    25Ôćĺ- **Custom auth instead of NextAuth** ÔÇö although `next-auth` is listed as a dependency, the actual authentication system is fully custom: password hashing uses Node.js `crypto.scryptSync`, session tokens are generated with `crypto.randomBytes(32)` and stored in the `Session` table, and session cookies are set as `HttpOnly` / `Secure` / `SameSite=Lax`.
    26Ôćĺ- **Multi-tenant RBAC** ÔÇö three user roles: `super_admin` (platform-wide), `tenant_admin` (tenant-wide), and `member` (tenant-scoped). All API routes enforce tenant isolation.

   115Ôćĺ## Step 3 ÔÇö Create Application in Coolify
   116Ôćĺ
   117Ôćĺ1. In Coolify, navigate to **Resources** Ôćĺ **New Resource** Ôćĺ **Application**
   118Ôćĺ2. Select **GitHub** (or GitLab) as the source
   119Ôćĺ3. Authenticate with your GitHub account and select the repository containing this code
   120Ôćĺ4. Configure the build settings:
   121Ôćĺ
   122Ôćĺ| Setting | Value |
   123Ôćĺ|---------|-------|
   124Ôćĺ| **Build Pack** | `Dockerfile` |
   125Ôćĺ| **Dockerfile Location** | `Dockerfile` (root of the repo) |
   126Ôćĺ| **Port** | `3000` |
   127Ôćĺ| **Branch** | `main` |
   128Ôćĺ
   129Ôćĺ> **Important ÔÇö Dockerfile Required:** The project uses a multi-stage Docker build. If a `Dockerfile` does not yet exist in the repo root, create one (see the Dockerfile section below). Coolify will use it to build the container image.
   130Ôćĺ
   131Ôćĺ### Dockerfile (create if not present)
   132Ôćĺ
   133Ôćĺ```dockerfile
   134Ôćĺ# ---- Stage 1: Dependencies ----
   135ÔćĺFROM oven/bun:1 AS deps
   136ÔćĺWORKDIR /app
   137ÔćĺCOPY package.json bun.lock ./
   138ÔćĺRUN bun install --frozen-lockfile
   139Ôćĺ
   140Ôćĺ# ---- Stage 2: Build ----
   141ÔćĺFROM oven/bun:1 AS builder
   142ÔćĺWORKDIR /app
   143ÔćĺCOPY --from=deps /app/node_modules ./node_modules
   144ÔćĺCOPY . .
   145ÔćĺRUN bunx prisma generate
   146ÔćĺRUN bun run build
   147Ôćĺ
   148Ôćĺ# ---- Stage 3: Production ----
   149ÔćĺFROM oven/bun:1 AS runner
   150ÔćĺWORKDIR /app
   151Ôćĺ
   152ÔćĺENV NODE_ENV=production
   153ÔćĺENV PORT=3000
   154Ôćĺ
   155Ôćĺ# Create uploads directory
   156ÔćĺRUN mkdir -p /app/uploads
   157Ôćĺ
   158Ôćĺ# Copy standalone output and public assets
   159ÔćĺCOPY --from=builder /app/.next/standalone ./
   160ÔćĺCOPY --from=builder /app/.next/static ./.next/static
   161ÔćĺCOPY --from=builder /app/public ./public
   162ÔćĺCOPY --from=builder /app/prisma ./prisma
   163Ôćĺ
   164Ôćĺ# Copy node_modules for Prisma CLI at runtime
   165ÔćĺCOPY --from=deps /app/node_modules/.prisma ./node_modules/.prisma
   166ÔćĺCOPY --from=deps /app/node_modules/@prisma ./node_modules/@prisma
   167ÔćĺCOPY --from=deps /app/node_modules/prisma ./node_modules/prisma
   168Ôćĺ
   169Ôćĺ# Copy seed script
   170ÔćĺCOPY --from=builder /app/prisma/seed.ts ./prisma/seed.ts
   171ÔćĺCOPY --from=builder /app/src ./src
   172Ôćĺ
   173ÔćĺEXPOSE 3000
   174Ôćĺ
   175Ôćĺ# Entrypoint runs Prisma db push on every startup, then seeds if SEED_DB=true
   176ÔćĺCOPY <<'ENTRYPOINT' /app/entrypoint.sh
   177Ôćĺ#!/bin/sh
   178Ôćĺset -e
   179Ôćĺecho ">>> Running Prisma db push..."
   180Ôćĺbunx prisma db push --accept-data-loss 2>/dev/null || bunx prisma db push
   181Ôćĺecho ">>> Prisma schema synced."
   182Ôćĺif [ "$SEED_DB" = "true" ]; then
   183Ôćĺ  echo ">>> Seeding database..."
   184Ôćĺ  bun run prisma/seed.ts
   185Ôćĺ  echo ">>> Seed complete."
   186Ôćĺfi
   187Ôćĺecho ">>> Starting iSecurify GRC Platform on port ${PORT:-3000}..."
   188Ôćĺexec bun .next/standalone/server.js
   189ÔćĺENTRYPOINT
   190Ôćĺ
   191ÔćĺRUN chmod +x /app/entrypoint.sh
   192Ôćĺ
   193ÔćĺCMD ["/app/entrypoint.sh"]
   194Ôćĺ```
   195Ôćĺ
   196Ôćĺ> **What the entrypoint does:**
   197Ôćĺ> 1. Runs `prisma db push` to sync the schema to MySQL on every container startup
   198Ôćĺ> 2. If `SEED_DB=true`, runs the seed script to create demo data
   199Ôćĺ> 3. Starts the Next.js standalone server

   203Ôćĺ## Step 4 ÔÇö Environment Variables in Coolify
   204Ôćĺ
   205ÔćĺIn Coolify, navigate to your **Application** Ôćĺ **Environment Variables** panel and add each of the following:
   206Ôćĺ

   210Ôćĺ| `NEXTAUTH_SECRET` | _(generate below)_ | Required by the `next-auth` package even though custom auth is used. Generate with: `openssl rand -base64 32` |

   214Ôćĺ| `PORT` | `3000` | The port the standalone server listens on. Must match the Coolify port config. |
   215Ôćĺ| `UPLOAD_PATH` | `/app/uploads` | Filesystem path where evidence uploads are stored inside the container. |
   216Ôćĺ| `SEED_DB` | `true` | **First deploy only.** Set to `true` to seed the database with demo users, tenants, frameworks, and controls. Remove or set to `false` after first successful deploy. |

   218Ôćĺ### Generating a Secure NEXTAUTH_SECRET
   219Ôćĺ
   220Ôćĺ```bash
   221Ôćĺopenssl rand -base64 32
   222Ôćĺ```
   223Ôćĺ
   224ÔćĺCopy the output and paste it as the value for `NEXTAUTH_SECRET`. Example output:
   225Ôćĺ```
   226ÔćĺaB3dE7fG9hJ2kL5mN8pQ1rS4tU6vW0xY3zA6cD9eF2gH5iK8jL1mN4pQ7r==
   227Ôćĺ```
   228Ôćĺ
   229Ôćĺ### Environment Variable Notes
   230Ôćĺ
   231Ôćĺ- **DATABASE_URL host**: The hostname in DATABASE_URL must match the Coolify service name of your MySQL resource. If your MySQL service is named `mysql` in Coolify, use `mysql` as the host. If named `mysql-isecurify-db`, use that. Coolify's internal DNS resolves service names to container IPs.
   232Ôćĺ- **NEXTAUTH_SECRET**: Although the platform uses a custom session-based auth system (not NextAuth handlers), the `next-auth` package is imported as a dependency and may throw errors if `NEXTAUTH_SECRET` is not set. Always include it.
   233Ôćĺ- **SEED_DB**: This env var is checked by the Docker entrypoint script. When `true`, the seed script (`prisma/seed.ts`) runs after `prisma db push`. The seed creates a Super Admin user, a demo tenant (Acme Corporation), sample frameworks (SOC 2, ISO 27001), controls, and audit data.

   237Ôćĺ## Step 5 ÔÇö Persistent Storage (Uploads)
   238Ôćĺ
   239ÔćĺEvidence files uploaded through the platform (screenshots, documents, compliance artifacts) are stored on the container filesystem. Without persistent storage, these files are lost on every redeploy.
   240Ôćĺ
   241Ôćĺ### Mount a Volume in Coolify
   242Ôćĺ
   243Ôćĺ1. Navigate to your **Application** in Coolify
   244Ôćĺ2. Go to the **Storages** (or **Volumes**) tab
   245Ôćĺ3. Click **Add Storage**
   246Ôćĺ4. Set the following:
   247Ôćĺ
   248Ôćĺ| Setting | Value |
   249Ôćĺ|---------|-------|
   250Ôćĺ| **Container Path** | `/app/uploads` |
   251Ôćĺ| **Volume Name** | Coolify will auto-generate a name (e.g., `isecurify-uploads`) |
   252Ôćĺ| **Type** | Volume (not bind mount) |
   253Ôćĺ
   254Ôćĺ5. Click **Save**
   255Ôćĺ
   256ÔćĺThe Dockerfile already creates `/app/uploads` during the build, and the Coolify volume mount ensures this directory persists across container restarts, rebuilds, and redeployments.
   257Ôćĺ
   258Ôćĺ> **How uploads work:** When a user uploads evidence via the API (`POST /api/evidence`), the server saves the file to `/app/uploads/{controlId}/{filename}` and stores a relative `filePath` (e.g., `uploads/ctrl-abc123/evidence.pdf`) in the database. The volume mount ensures these files survive container lifecycle events.

   287Ôćĺ## Step 7 ÔÇö First Deployment
   288Ôćĺ
   289Ôćĺ### Deploy with Seed Data
   290Ôćĺ
   291Ôćĺ1. Ensure all environment variables from **Step 4** are configured
   292Ôćĺ2. Ensure `SEED_DB=true` is set in the environment variables
   293Ôćĺ3. Ensure the MySQL service is running in Coolify (check its status in Resources)
   294Ôćĺ4. In Coolify, navigate to your **Application** Ôćĺ **Deployments** tab
   295Ôćĺ5. Click **Deploy Now**
   296Ôćĺ
   297Ôćĺ### What Happens During First Deploy
   298Ôćĺ
   299Ôćĺ1. Coolify pulls your code from GitHub
   300Ôćĺ2. Docker builds the image using the Dockerfile
   301Ôćĺ3. The container starts and runs the entrypoint script:
   302Ôćĺ   - `prisma db push` syncs the schema to MySQL (creates all tables)
   303Ôćĺ   - `bun run prisma/seed.ts` creates demo data (since `SEED_DB=true`)
   304Ôćĺ   - Next.js standalone server starts on port 3000
   305Ôćĺ4. Coolify's reverse proxy routes traffic to the container
   306Ôćĺ
   307Ôćĺ### After First Deploy
   308Ôćĺ
   309Ôćĺ1. Verify the app loads at your domain
   310Ôćĺ2. Log in with the default credentials (see below)
   311Ôćĺ3. **Immediately change the default passwords**
   312Ôćĺ4. Go back to **Environment Variables** in Coolify
   313Ôćĺ5. Remove `SEED_DB=true` or set it to `false`
   314Ôćĺ6. **Redeploy** to ensure the seed doesn't run again on future startups
   315Ôćĺ
   316Ôćĺ> **Why remove SEED_DB?** The seed script uses `upsert` operations, so it won't duplicate data. However, it will reset passwords back to defaults if left enabled. Always remove it after the first successful deploy.

   389Ôćĺ
   390Ôćĺ### Manual Deploy

   393Ôćĺ2. In Coolify, navigate to your Application Ôćĺ Deployments
   394Ôćĺ3. Click **Deploy Now**
   395Ôćĺ
   396Ôćĺ### Schema Changes
   397Ôćĺ
   398ÔćĺWhen you update `prisma/schema.prisma`:
   399Ôćĺ
   400Ôćĺ1. Commit and push the schema change to GitHub
   401Ôćĺ2. The Docker build runs `prisma generate` during the build stage
   402Ôćĺ3. On container startup, the entrypoint runs `prisma db push` ÔÇö this automatically applies schema changes to MySQL
   403Ôćĺ4. No manual migration commands needed
   404Ôćĺ
   405Ôćĺ> **If you need to accept data loss** (e.g., dropping a column), the entrypoint uses `--accept-data-loss` flag by default. Review schema changes carefully before deploying.

   409Ôćĺ## File Uploads
   410Ôćĺ
   411Ôćĺ### How Uploads Work
   412Ôćĺ
   413Ôćĺ1. **Client** sends a `POST` request to `/api/evidence` with a file
   414Ôćĺ2. **Server** saves the file to `/app/uploads/{controlId}/{uuid}-{filename}`
   415Ôćĺ3. **Database** stores the relative file path (e.g., `uploads/ctrl-soc2-01/a1b2c3-report.pdf`)
   416Ôćĺ4. **Volume** at `/app/uploads` persists the files across container lifecycle events
   417Ôćĺ
   418Ôćĺ### Upload Storage Details
   419Ôćĺ
   420Ôćĺ| Aspect | Detail |
   421Ôćĺ|--------|--------|
   422Ôćĺ| **Container Path** | `/app/uploads` |
   423Ôćĺ| **Volume Type** | Docker named volume (managed by Coolify) |
   424Ôćĺ| **Persistence** | Survives container restarts, rebuilds, and redeployments |
   425Ôćĺ| **Access** | Files are served through the API (`GET /api/evidence`) or directly if public assets are configured |
   426Ôćĺ| **Cleanup** | No automatic cleanup ÔÇö manage disk space by archiving old evidence |
   427Ôćĺ
   428Ôćĺ### Backing Up Uploads
   429Ôćĺ
   430Ôćĺ```bash
   431Ôćĺ# From the host machine, copy the volume contents to a backup location
   432Ôćĺdocker cp coolify-isecurify-app-1:/app/uploads ./uploads-backup-$(date +%Y%m%d)
   433Ôćĺ```
   434Ôćĺ
   435ÔćĺOr set up a cron job to periodically sync to S3-compatible storage:
   436Ôćĺ
   437Ôćĺ```bash
   438Ôćĺ# Using AWS CLI (install awscli first)
   439Ôćĺaws s3 sync /var/lib/docker/volumes/isecurify-uploads/_data s3://your-bucket/isecurify-uploads/
   440Ôćĺ```

   444Ôćĺ## Troubleshooting
   445Ôćĺ
   446Ôćĺ| Problem | Solution |
   447Ôćĺ|---------|----------|
   448Ôćĺ| **`DATABASE_URL not found` error at startup** | Check Coolify Ôćĺ Application Ôćĺ Environment Variables. The `DATABASE_URL` must be present and the `mysql://` protocol prefix must be included. |
   449Ôćĺ| **`Can't reach database server` / connection refused** | Verify the MySQL service name in Coolify matches the hostname in `DATABASE_URL`. If your MySQL service is named `mysql-isecurify`, use `mysql-isecurify` (not `localhost` or `127.0.0.1`). |
   450Ôćĺ| **`Table 'isecurify.User' doesn't exist`** | Prisma `db push` didn't run successfully. Check the container startup logs in Coolify Ôćĺ Application Ôćĺ Logs. Look for errors from `prisma db push`. |
   451Ôćĺ| **Build fails on `bun install`** | Ensure `bun.lock` is committed to the Git repository. The Dockerfile uses `--frozen-lockfile` which requires the lockfile. Run `bun install` locally and commit the generated `bun.lock`. |
   452Ôćĺ| **Build fails on `prisma generate`** | Check that `prisma/schema.prisma` is valid. Run `bunx prisma validate` locally. Also ensure the datasource `provider` is set to `"mysql"` (not `"sqlite"`) for production. |
   453Ôćĺ| **Uploads not persisting across deploys** | Verify that the `/app/uploads` volume is mounted in Coolify Ôćĺ Application Ôćĺ Storages. If the volume is missing, uploaded files exist only inside the ephemeral container filesystem. |
   454Ôćĺ| **`NEXTAUTH_SECRET` error in logs** | Set the `NEXTAUTH_SECRET` environment variable in Coolify, even though the platform uses custom auth. The `next-auth` package checks for this variable during initialization. Generate one with `openssl rand -base64 32`. |
   455Ôćĺ| **Login redirect loop or cookies not set** | Ensure SSL is enabled (HTTPS). Session cookies use the `Secure` flag and will not be sent over plain HTTP. Check Coolify Ôćĺ Domains Ôćĺ SSL Certificate is active. |
   456Ôćĺ| **Seed data not created on first deploy** | Verify `SEED_DB=true` is set in environment variables. Check the entrypoint logs for "Seeding database..." output. Also ensure `DATABASE_URL` is correct so `prisma db push` succeeds before the seed runs. |
   457Ôćĺ| **Page shows 502 Bad Gateway** | The container may have crashed. Check Coolify Ôćĺ Application Ôćĺ Logs for startup errors. Common causes: database connection failure, missing env vars, or build errors. |
   458Ôćĺ| **API returns 401 Unauthorized** | Your session may have expired. Clear cookies and log in again. If the issue persists, check that the `Session` table exists in MySQL and session tokens are being created correctly. |

   463Ôćĺ
   464Ôćĺ### MySQL Database Backups
   465Ôćĺ
   466Ôćĺ1. Navigate to Coolify Ôćĺ **MySQL Resource** Ôćĺ **Backups** tab
   467Ôćĺ2. Enable **Scheduled Backups**
   468Ôćĺ3. Configure the schedule (e.g., daily at 02:00 UTC)
   469Ôćĺ4. Set a retention policy (e.g., keep last 30 days)
   470Ôćĺ5. Optionally configure remote backup storage (S3, SFTP, etc.)
   471Ôćĺ
   472Ôćĺ### Manual MySQL Backup
   473Ôćĺ
   474ÔćĺFrom Coolify Terminal on the MySQL service:
   475Ôćĺ
   476Ôćĺ```bash
   477Ôćĺmysqldump -u isecurify_user -p isecurify > /var/lib/mysql/backups/isecurify-$(date +%Y%m%d).sql
   478Ôćĺ```
   479Ôćĺ
   480Ôćĺ### File Upload Backups
   481Ôćĺ
   482ÔćĺThe `/app/uploads` volume contains all user-uploaded evidence files. Back these up regularly:
   483Ôćĺ
   484Ôćĺ```bash
   485Ôćĺ# From the host machine
   486Ôćĺdocker cp <container_id>:/app/uploads ./uploads-backup
   487Ôćĺ```
   488Ôćĺ
   489ÔćĺFor automated backups, consider:
   490Ôćĺ
   491Ôćĺ- **S3 sync** with a cron job on the host
   492Ôćĺ- **Restic** or **Borg** for encrypted, deduplicated backups
   493Ôćĺ- **Coolify's built-in volume backup** (if available in your Coolify version)
